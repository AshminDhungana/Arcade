# Arcade Agent Setup Guide

The Arcade Agent runs on each client (gaming) PC as an Electron-based kiosk overlay. It connects to the server via WebSocket, handles session state locally, and manages the kiosk overlay.

## Quick Start

1. Run the **Launcher Setup Wizard** on the server PC to generate `agent_secrets` for each seat.
2. Copy the generated `agent.config.json` to each client PC, next to the agent executable.
3. (Linux/macOS) Set file permissions: `chmod 600 agent.config.json`
4. Start the Arcade Agent. It will connect to the server automatically.

---

## `agent.config.json`

`agent.config.json` is a JSON file read at agent startup. It tells the agent which server to connect to, which seat it represents, and how to authenticate.

### Schema

```json
{
  "server_url": "ws://192.168.1.100:8000",
  "seat_id": "seat_001",
  "agent_secret": "c9a1b2c3d4e5f6...",
  "override_code_hash": null,
  "reconnect_max_seconds": 60,
  "health_interval_seconds": 60
}
```

### Required Fields

| Field | Type | Description |
|-------|------|-------------|
| `server_url` | `string` | WebSocket URL of the Arcade server. Must start with `ws://` or `wss://`. |
| `seat_id` | `string` | Seat identifier matching the server's `arcade.config.json` (e.g. `seat_001`). |
| `agent_secret` | `string` | Shared secret for WebSocket authentication. Generated by the setup wizard (64-char hex). |

### Optional Fields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `override_code_hash` | `string \| null` | `null` | Argon2id hash of the staff override PIN. If set, staff can temporarily disable the kiosk overlay. |
| `reconnect_max_seconds` | `number` | `60` | Maximum reconnect delay cap in seconds. |
| `health_interval_seconds` | `number` | `60` | Interval between health metric reports to the server, in seconds. |

### Where to Place the File

- **Production (packaged):** Next to the agent executable (`.exe` on Windows, `.app` on macOS, binary on Linux).
- **Development:** The agent falls back to the current working directory if the file is not found next to the executable.

### File Permissions

The `agent_secret` is a sensitive credential. On Linux and macOS, restrict the file so only the owner can read it:

```bash
chmod 600 agent.config.json
```

> **Security:** Never commit `agent.config.json` to version control. It is already `.gitignore`d.

## Obtaining Config Values

All values are generated during the **Launcher Setup Wizard** on the server PC:

1. **Launch the Arcade Launcher** on the server PC.
2. **Activate the license** (place `license.key` in the app directory).
3. The **Setup Wizard** runs automatically (first launch only):
   - Set cafe name, server host, and port.
   - Create admin and cashier staff accounts.
   - Define the number of seats.
   - For each seat, a unique `agent_secret` is generated via `secrets.token_hex(32)`.
4. The wizard writes `arcade.config.json` on the server.
5. **Deploy `agent.config.json`** to each client PC:
   - Copy the generated `agent.config.json` for that seat.
   - Update `server_url` to point at the server PC's IP address.
   - Place the file next to the agent executable.

## Config Loading Behaviour

On startup, the agent:

1. **Determines the config file path:**
   - First, looks next to `process.execPath` (the agent executable).
   - Falls back to `process.cwd()` (current working directory) in development.
2. **Reads and parses** `agent.config.json`.
3. **Validates** all required fields and types.
4. **Applies defaults** for optional fields.
5. If any step fails, an error dialog is shown and the agent exits.

### Production

```
agent.exe
agent.config.json   <-- read from here
```

### Development

```
agent/
├── src/
├── dist/
└── agent.config.json   <-- falls back to cwd
```

## Auto-Start Configuration

The agent supports automatic start on system boot via the platform abstraction layer:

- **Windows:** Registry entry `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`
- **macOS:** LaunchAgent plist at `~/Library/LaunchAgents/com.arcade.agent.plist`
- **Linux:** systemd service or `.desktop` file in `~/.config/autostart/`

Enable via the dashboard (Settings → Agent → Auto-Start) or programmatically via `platform.enableAutoStart()`.

## Troubleshooting

### "agent.config.json not found"

- Ensure the file is named exactly `agent.config.json` (case-sensitive on Linux/macOS).
- Ensure it is in the same directory as the agent executable.
- In development, ensure it is in the current working directory.

### "server_url must start withtis ws:// or wss://"

- The `server_url` must use the WebSocket protocol, not HTTP.
- Correct: `ws://192.168.1.100:8000` or `wss://10.0.0.1:8443`
- Incorrect: `http://192.168.1.100:8000`

### "seat_id is required"

- The `seat_id` must match the seat identifier in the server's `arcade.config.json`.
- It is case-sensitive.

### "agent_secret is required"

- The `agent_secret` must be generated by the setup wizard.
- It is a 64-character hexadecimal string.
- Do not confuse it with the server's `jwt_secret`.

### Agent fails to connect after config update

- The agent reads the config **once at startup**. Changes require a restart.
- Ensure the server is reachable from the agent PC (ping the server IP).
- Check that the server's WebSocket endpoint is active (`/ws/agent/{seat_id}`).

### Config not being read from the expected location

- The agent logs the resolved config path on startup. Check the logs.
- In development, the current working directory (`cwd`) is used as a fallback.
